Finalization Latency (TTFS): bounds, factors, and where the wins are

Scope: the fork configuration (Step 7d) — server-side continuous context + short debounce + speculative disposable fork. Legend: measured · modeled · reasoned (pending confirmation). (Revised per the Codex technical-accuracy review: §3 floor recast as a modeled-formula floor, not irreducible; §6 blank/entropy downgraded to "requires enabling supported decode instrumentation"; flush/transport wording fixed; discard-safety clarified as pre-emit only.)

3. Floors: modeled-formula vs post-decision wall-clock

Two different floors — do not conflate them.
(a) Modeled budget floor ≈ ~175 ms = rc1 (160, modeled constant) + finalize flush (~14), under the locked additive formula. This is a conservative accounting floor, not a physical one.
(b) True post-endpoint-decision wall-clock floor ≈ flush + transport + quantization (~tens of ms). Once the system decides to finalize, the fork supplies the last chunk's right-context as synthetic (R+1)*shift zeros (faster-than-wallclock) — it does not wait 160 ms of real audio. So rc1 is not an irreducible post-decision cost.
The real limiter is the endpoint decision itself: you cannot finalize until you have enough evidence that speech ended. With a perfect oracle that evidence is ~0; with any realistic signal it is whatever observation window that signal needs — which typically overlaps rc1. So the achievable target is "collapse the ~350 ms of stacked static waits down to the evidence window of a good endpoint signal," not "hit a fixed 175 ms."

4. What we are doing now (Step 7d, fork)

  • Short 150 ms server debounce instead of the 2500 ms analytical hold (cc7b/cc7c).
  • Speculative disposable fork: on debounce expiry we finalize on a deep-clone fork; the live ASR context is untouched. If speech resumes before the debounce fires, vad_start cancels the pending finalize and the fork is discarded — protecting the live context from cold-reset damage. Caveat: this is harmless only pre-emit; once a speculative final is emitted it is concatenated by the append-only collector and cannot be revoked (see §6).
  • Always-append-only emission keeps the multi-segment transcript duplication-free.
  • Preliminary measured budget p95 ~326 ms (PASS < 400 ms): endpoint_wait ~151 + rc1 160 (modeled) + flush ~13 + transport ~0.2; lock-wait ~0 (sequential).

5. How the 150 ms debounce was chosen

From the Step-1 offline Silero VAD preflight (stop→next-start gap distribution: min 154 ms, p50 380 ms, p95 800 ms) and the locked <400 ms budget allowance (400 − rc1 160 − flush ~14 − margin ≈ ~150). 150 ms is the largest value that fits the budget and sits just below the minimum observed real intra-utterance gap (154 ms), so a true resume almost never lands inside the window — and when it does (pre-emit), the fork-discard absorbs it. It is a budget-bounded, data-informed static choice. Update (§8): we later established the 150 ms is purely a VAD-cancellation safety window (audio is held aside, not fed to the ASR), so for a definitive end-of-turn trigger it can be set to 0 with no WER change (99/100 byte-identical) — a measured −152 ms TTFS win.

6. Can we do better than a static wait? (the real opportunity)

The debounce only proxies "did the speaker actually stop, or is this a mid-utterance pause?" A fixed timer is a blunt proxy for signals that can be derived from the ASR itself. Important constraint: today the service calls conformer_stream_step(...) and receives text + hypotheses + cache state — not per-frame RNNT blank probabilities or joint logits/entropy. Those are not "already in hand"; exposing them needs NeMo's supported decode instrumentation (preserve_alignments / preserve_frame_confidence / confidence_method_cfg via change_decoding_strategy) — a config change, no retraining and no framework edit, but its viability on this cache-aware streaming path is a feasibility question to test, not an assumption.

Honest framing of the win — what a false-early-fire actually costs. The configured decode is greedy RNNT (strategy: greedy), which is append-only on the committed token sequence: it never un-emits a token. So a false-early-fire does not rewrite arbitrary earlier text, and once a frame has its full ~160 ms right context its decoded tokens are stable. The real cost is therefore bounded and local, not a global rewrite: (a) the still-provisional last ≤rc1 (~160 ms) tail gets flushed against the fork's synthetic silence instead of the real continuing speech, so the last word or two before the pause can be committed wrong; (b) a possible spurious seam token at the fake boundary; (c) the segmentation/restart family. All three are frozen by the append-only collector (irrevocable post-emit; a pre-emit discard is still free). This means a stability signal that waits until the tail has settled past the rc1 window is genuinely safe from the rewrite concern — but the seam/provisional-tail cost is still WER-gated. The earlier characterization ("~40% non-prefix transitions ⇒ earlier text is rewritten") is not evidence of beyond-rc1 rewrite on greedy; it conflates the provisional ≤rc1 tail, interim-vs-flush tail, and detok/casing/punctuation churn (token-prefix-stable, semantic-WER-tolerant). Whether any genuine edit to a token older than its right-context window ever occurs on this path is an explicit measurement in the next-stage plan, not an assumption.

7. Empirical EOU result on this checkpoint (2026-05-20 follow-up)

The "speculation" table in §6 has been tested. Verdict: NO-GO for the voice-agent end-of-turn use case on nvidia/nemotron-speech-streaming-en-0.6b. The warm200 shippable from the parent project remains the recommended production configuration.

What we built and measured. Env-gated probe instrumentation (NEMOTRON_EOU_PROBE=1 with preserve_alignments, preserve_frame_confidence, and tsallis-entropy confidence_method_cfg installed via change_decoding_strategy at startup — the §6 "supported decode instrumentation" path is feasible: it is argmax-invariant to baseline and default-off byte-identical to fork/warm200). Per-chunk state snapshots (NEMOTRON_EOU_SNAPSHOT_DIR=...) so any operating point can be replayed offline. ROC sweep across the three signal families from §6 (hypothesis stability, blank-run, normalized confidence) at 28 operating points, on 100 LibriSpeech sessions (duration-representative of the full 1000).

What we found. The 28-point viability table (latency p95 ≤ {0, 50, 100, 150, 200} ms crossed with false-early-fire rate ≤ {0.5%, 1%, 2%, 5%}) had zero matches. Each signal has the same bimodal shape: at low K it fires constantly on natural in-utterance pauses (97-100% false-fire at K≤3), at high K it doesn't fire at all in most sessions (95% never-fired at K≥60 encoder frames). The hyp_unchanged K and blank_run K signals fire on identical chunks at every K.

Why (mechanism, not tuning). Voice-agent end-of-turn detection must discriminate inter-sentence pauses (~280 ms in our data) from end-of-turn pauses (~2 s). The three signal families are all duration-thresholded on decoder-emission counters; they cannot do anything a calibrated pause-duration threshold cannot. The moment K is set large enough to ignore inter-sentence pauses, the wallclock latency equals Silero's. The §6 prediction "a stability signal that waits until the tail has settled past the rc1 window is genuinely safe from the rewrite concern" was confirmed (Step 2b's rc1-stability analysis: greedy RNNT is append-only on committed tokens; the rewrite concern was overstated in earlier analysis). What the rc1-stability finding does NOT do, on this checkpoint: give us a non-duration signal to discriminate end-of-turn from inter-sentence-pause.

Architecturally, the silence-pump door is open. The server has no per-chunk rate-limiting (_handle_audio_locked processes back-to-back, ~10 ms GPU per 160 ms chunk ≈ 16× compute-vs-realtime ceiling). The Step-7d fork-flush already feeds (R+1) × shift_frames × hop_samples = 320 ms of synthetic zeros through conformer_stream_step at faster-than-wallclock pace. Extending this to longer synthetic silence is feasible without any framework or decoder-mode change. It just doesn't help the end-of-turn problem: even an unbounded-K synthetic-silence pump cannot synthesize a non-duration signal that this decoder doesn't expose.

8. Finalize debounce can go to ZERO (2026-05-20, measured)

The §5 framing of the 150 ms debounce as a budget-bounded settle margin was incomplete. Tracing the actual server code (Codex-verified): during the debounce window the incoming audio is held aside in continuous_post_stop_audio and NOT fed to the ASR; the fork-flush runs from the parent state as of vad_stop regardless of the debounce value. So the 150 ms is purely a VAD-cancellation safety window ("wait briefly to see if the speaker resumes before committing the finalize"), not time the model needs to settle.

Offline replay (silence_pad_replay.py) over 100 sessions confirmed the fork's existing (R+1)*shift = 320 ms synthetic pad already converges. On the 33 single-vad_stop baseline-comparable sessions (where the production transcript is a clean single-segment reference), replaying from the vad_stop snapshot reproduced production at median edit-distance 0, p95 3.4 chars (CUDA/cuFFT nondeterminism); longer pads (480/640/800 ms) did not improve this, while 1200 ms actively degraded (out-of-distribution synthetic input). (The remaining 67 sessions are multi-vad_stop continuous-mode runs whose DB baseline is the concatenated multi-segment aggregate, so the single-fork replay's larger edit-distance there is expected, not a defect.) Measured 100-sample A/B (silence_0 vs silence_150): normalized word-error-rate vs ground truth identical (4.286% both, Δ 0.0000 pp; computed directly against ground truth — the Gemini/Claude semantic judge is even more tolerant), transcripts 99/100 byte-identical (the one diff a single inserted period), at TTFB 215 ms vs 367 ms — a 152 ms reduction.

Result. For a use case with a definitive end-of-turn trigger (e.g. Smart Turn) — or where the segmentation cost of an occasional early finalize is acceptable — NEMOTRON_FINALIZE_SILENCE_MS=0 drops TTFS p95 from ~366 ms to ~215 ms with no transcript change. The fork's unfork-safe state preservation (deep clone; FORK_ASSERT-verified) means a false-early fire is recoverable: emit the finalize now, keep processing the retained context, continue on the next utterance. The 150 ms remains the right setting only when the trigger itself is unreliable (Silero pause vs end-of-turn ambiguity) and the safety window is doing real cancellation work.

Ship-gate confirmation (full 1000, measured 2026-05-20). A full-1000 run of silence0_warm200 at concurrency 12, scored by the Claude semantic-WER judge: pooled semantic WER 1.95% vs warm200's 2.07% / fork's 2.08% — non-inferior (the ~0.1 pp delta is within judge + sampling noise; the baselines were scored in a separate judge run, so this is not a paired comparison). TTFB p95 247 ms under 12-way concurrency vs warm200's 366 ms single-session — a ~119 ms reduction. Both ship-gate criteria met: WER non-inferior to warm200 AND latency well inside the 400 ms budget. silence0_warm200 is the recommended production configuration for a definitive-trigger use case, superseding bare warm200.

9. Concurrency / capacity (2026-05-20, measured — R4 caveat closed)

The server does no request batching — every conformer_stream_step is serialized through one global inference_lock, model batch_size=1 by design (the batch API corrupts the cache-aware streaming state). A realtime concurrent sweep (concurrency_test.py, N ∈ {1,4,8,12,16,20,24} live sessions, production silence_0 + warm200 config) found:

N concurrentbyte-accuracyserver finalize p95 (vad_stop→final)end-to-end TTFS ≈ +200 ms Silero

16100% exact33 ms~233 ms — in budget, edge

24100% exact2082 ms~2280 ms — server behind realtime

Correctness is perfect at every N (max edit-distance 0 vs single-session baseline, FORK_ASSERT clean) — the serialized-inference + per-session-cache design degrades gracefully: latency only, never correctness, even when overwhelmed. Capacity: ~12 concurrent live sessions per RTX 5090 comfortably in-budget, ~16 at the edge; the single inference_lock saturates at ~N=16–18. Scales horizontally (capacity = per-GPU-ceiling × instances). Warm-up (warm200) connect contention is negligible (ready p95 ≤ 52 ms at N=24). The unexplored lever for higher per-GPU throughput is vLLM-style continuous batching on the cache-aware streaming path — significant work, not yet attempted. (Scope: measured on one RTX 5090, this checkpoint, a 24-audio realtime sweep with tightly-grouped randomized starts; production hardware/traffic could shift the knee.)

10. Honest caveats

  • Two floors, not one. ~175 ms is a modeled formula floor; the true post-decision wall-clock floor is lower (synthetic rc1) and the binding limit is the endpoint-evidence window of whatever signal triggers finalize.
  • Measured vs modeled vs reasoned. endpoint_wait/flush/transport/lock are measured; rc1 (160 ms) is a modeled constant; the rc1↔wait overlap and the floor reasoning are reasoned, to be cross-checked by the run-end observer-TTFS.
  • Greedy append-only / rc1-stability is reasoned, then measured. "Committed tokens are not rewritten beyond their rc1 window on greedy" follows from the decode algorithm, but the next-stage plan instruments per-chunk token sequences to quantify any genuine older-than-rc1 edit (expected ≈0) rather than assume it — that result sets how aggressive the endpoint trigger can safely be.
  • Preliminary. Budget p95 ~326 ms is from ~499/1000 samples; per-term distributions are tight, so it is unlikely to move materially.
  • Concurrency (R4) — now measured (§9). Correctness is byte-exact to N=24; latency is flat to ~12 concurrent sessions/GPU, knee at ~16–18. The earlier "not captured" caveat is closed.
  • WER is decided separately. Low latency only counts if the always-append-only fork WER lands in the Phase-G/oracle region — that gate is pending the judge/score at run completion.
